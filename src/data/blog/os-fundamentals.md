## 一、什么是操作系统

操作系统（Operating System）是计算机系统中最基础、最核心的软件。站在不同角度，它扮演着三种不同的角色：

**（1）系统资源的管理者**

操作系统管理着计算机的所有资源，既包括硬件资源（CPU、内存、硬盘、网卡等），也包括软件资源（文件、进程、程序等）。它像一个中央调度中心，决定谁用什么资源、用多久、什么时候释放。如果没有操作系统统一管理，各个程序各自为政，整个系统很快就会混乱崩溃。

**（2）用户和软件的接口与环境**

操作系统为上层提供了方便的交互方式。对于普通用户来说，我们通过图形界面、命令行、触摸屏等方式和计算机交互；对于应用程序来说，操作系统提供了一系列标准化的系统调用和 API，让程序不用关心底层硬件的细节就能完成各种操作。这种标准化极大地降低了软件开发的复杂度。

**（3）最接近硬件的一层软件**

在计算机的层次结构中，操作系统处于硬件之上、应用程序之下。它是唯一可以直接和硬件对话的软件层——它知道怎么读写硬盘上的某个扇区，知道怎么控制 CPU 切换任务，知道怎么驱动网卡发送数据包。而其它的所有应用程序，都必须通过操作系统才能间接地操作硬件。

<span style="color: red;">

**补充理解——操作系统的层次定位：**

```
应用程序（浏览器、VSCode、游戏）
        ↓ 系统调用
    操作系统（管家）
        ↓ 驱动程序
   CPU / 内存 / 磁盘 / 网卡
```

没有 OS 的话，你写程序还得自己控制硬盘寻道、自己调度 CPU 时间片。有了 OS，你调一个 `write()` 系统调用，OS 帮你搞定硬件操作的全部细节。这就是操作系统存在的根本价值。

</span>

## 二、操作系统提供的功能

操作系统主要提供四大核心功能：

- **处理机管理**：决定哪个进程可以使用 CPU、用多长时间、什么时候切换。包括进程的创建、调度、同步和通信等，是整个操作系统的核心功能。
- **存储器管理**：负责内存的分配和回收，确保各个进程之间有内存保护（一个进程不能读写另一个进程的内存），并且在内存不够时能借助硬盘实现虚拟内存扩容。
- **文件管理**：将硬盘上的一大堆原始数据块组织成用户看得懂的文件和文件夹结构，提供创建、读写、删除、查找等文件操作。
- **设备管理**：管理各种输入输出设备（键盘、鼠标、显示器、打印机、磁盘等），提供统一的设备操作接口，处理不同设备之间的速度和操作差异。

这四大功能分工明确、互相配合，共同组成了操作系统的核心能力。

<span style="color: red;">

**补充——各功能展开：**

| 功能 | 核心问题 | 关键技术 |
|------|----------|----------|
| 进程管理 | 谁来用 CPU、用多久？ | 进程调度（RR、多级反馈队列）、进程同步（信号量、管程）、死锁处理 |
| 内存管理 | 多进程如何共享有限内存？ | 分页（Paging）、分段、虚拟内存、TLB 快表、页面置换（时钟算法） |
| 文件管理 | 磁盘上的 0/1 如何变成文件？ | inode 索引结构、目录组织、空闲空间管理（位图）、日志文件系统 |
| 设备管理 | 如何统一操作不同速度的设备？ | 中断驱动、DMA 直接内存访问、磁盘调度（电梯算法）、设备驱动层 |

</span>

## 三、操作系统的封装思想

操作系统最精妙的设计思想之一就是**封装**。

封装的意思是：操作系统把底层的硬件复杂性隐藏起来，只暴露出简单易用的服务接口。用户和应用程序只需要向操作系统发出指令，完全不需要关心硬件是怎么工作的。

举个例子：你保存一个文件，只需要在编辑器中点一下"保存"按钮。但在这背后，操作系统做了大量的工作——把数据写入内存缓冲区、寻址到硬盘的正确位置、控制磁头移动到对应扇区、把数据逐位写入磁性介质、验证写入的正确性……这些复杂的步骤全被 OS 封装在了 `write()` 这一个系统调用里面。

这种封装思想不仅让普通用户可以轻松使用计算机，也让程序员可以专注于实现业务逻辑，而不需要反复造底层硬件的轮子。它是整个计算机科学中分层设计和抽象思维的完美体现。

<span style="color: red;">

**补充——封装之外，OS 还有两个核心设计思想：**

**1. 虚拟化**：把物理资源抽象成虚拟资源。比如虚拟内存让每个进程以为自己独占 4GB 内存空间，实际上物理内存只有 8GB，20 个进程在共享使用。

**2. 并发与隔离**：多个程序"同时"运行（时间片轮转制造出的假象），但彼此之间完全隔离——一个进程崩了不会影响到其他进程，一个进程也看不到另一个进程的内存数据。这是现代 OS 稳定性的基石。

</span>

## 四、系统调用与用户态/内核态

<span style="color: red;">

封装的具体实现机制就是**系统调用**——它是用户程序和操作系统内核之间交互的唯一通道。

</span>

<span style="color: red;">

**系统调用的工作流程：**

比如 C 语言里写一个 `printf("hello")`：

```
printf("hello")           // C 标准库函数（用户态）
    ↓
write(1, "hello", 5)      // 系统调用（触发陷阱，用户态 → 内核态）
    ↓
内核代码把 "hello" 写到终端
```

常见系统调用：读文件（`open/read`）、创建进程（`fork`）、申请内存（`mmap`）、网络通信（`socket`）。在 Linux/WSL 里执行 `strace ls`，输出的每一行都是系统调用。

**用户态 vs 内核态：**

CPU 有两种运行模式，这是操作系统的安全基石：

| 用户态 | 内核态 |
|--------|--------|
| 普通程序运行的地方 | OS 内核运行的地方 |
| 不能直接访问硬件 | 能操作一切 |
| 不能执行特权指令 | 无限制 |

你的浏览器、VSCode 跑在用户态。当它想读写文件时，触发一个**陷阱（trap）**切换到内核态，OS 帮你干完活再切回来。这个切换有成本，所以频繁调系统调用会变慢。

</span>

---

## 五、操作系统内核类型

<span style="color: red;">

操作系统根据内核架构可以分为以下几种类型：

| 类型 | 代表系统 | 特点 | 优缺点 |
|------|----------|------|--------|
| **宏内核** | Linux、Windows（早期） | 全部功能（文件系统、网络、驱动）都跑在内核态 | 性能高（函数调用即可），但代码量大，一个模块出问题可能全系统崩溃 |
| **微内核** | Minix、QNX | 内核只做最基本的事（IPC、地址空间），文件系统、网络等作为用户进程运行 | 稳定性好（一个服务崩了不影响内核），但 IPC 通信开销大，性能不如宏内核 |
| **混合内核** | macOS(XNU)、Windows NT 实际架构 | 大部分功能在内核态（为性能），但也吸收了微内核的模块化设计思想 | 折中方案，兼顾性能和稳定性 |
| **外核** | MIT Exokernel 研究系统 | 内核只负责资源隔离和安全，把资源管理的决策权交给应用程序 | 极致灵活但极其复杂，主要用于研究 |

<span style="color: red;">

Linux 是典型的宏内核，Windows 虽然宣传上说是微内核，但实际上 NT 内核把图形、文件系统等大量组件都放在了内核态，本质上更接近混合内核。真正纯粹的微内核只有 Minix 和 QNX（用在汽车、医疗设备等对稳定性要求极高的场景）。

</span>

</span>

## 六、中断机制

<span style="color: red;">

中断是硬件通知 CPU "有事情需要处理" 的机制。它是操作系统响应外部事件的基石。

**中断的类型：**

- **硬件中断**：你按了键盘 → 键盘控制器发出中断 → CPU 停下当前任务 → OS 内核读取按键 → 回去继续
- **软件中断**：程序主动触发（系统调用本质上就是一个软件中断/trap）
- **异常**：程序出错（除零、访问非法地址等）

**为什么中断重要？**

中断让 OS 不需要一直轮询设备（"键盘有输入吗？硬盘读完了吗？"），而是**被动响应**——有事才处理，没事就让 CPU 安安静静干活。这极大地提高了 CPU 利用率。

中断处理程序（ISR）跑在内核态，要尽可能快——因为它打断了正常程序的执行，花太久会影响整个系统的响应。

</span>

---

## 七、进程管理

<span style="color: red;">

**什么是进程？**

进程 = 运行中的程序。程序和进程的区别：
- **程序**：硬盘上的可执行文件，是"死"的
- **进程**：程序加载到内存里运行，是"活"的

一个程序可以启动多次变成多个进程（比如你开了三个终端窗口）。

**进程在内存中的结构：**

```
高地址
┌──────────────┐
│    栈(stack)  │  ← 函数调用、局部变量（向下增长）
│     ↓        │
│              │  ← 空闲空间
│     ↑        │
│   堆(heap)   │  ← 动态分配的内存（malloc/new，向上增长）
├──────────────┤
│   数据段     │  ← 全局变量、静态变量
├──────────────┤
│   代码段(text)│  ← 程序的机器指令（只读）
└──────────────┘
低地址
```

**进程控制块（PCB）：**

每个进程在内核中都有一个 PCB，保存了 OS 管理它的所有信息：

| 字段 | 存什么 |
|------|--------|
| 进程 ID（PID） | 唯一的编号 |
| 程序计数器（PC） | 当前执行到哪条指令 |
| CPU 寄存器 | 被切换时需要保存的"现场" |
| 内存信息 | 页表指针 |
| I/O 状态 | 打开的文件列表 |
| 优先级 | 调度器用 |

**上下文切换**：CPU 从进程 A 切换到进程 B 时，先保存 A 的 PCB（寄存器和 PC 的值），再加载 B 的 PCB。这个操作没有"生产性"工作，纯粹是开销。切换太频繁 = 大量 CPU 时间浪费在切换本身上。

**进程的五种状态：**

```
     创建 → 就绪 → 运行 → 终止
              ↑      ↓
              └─等待←┘
```

- **创建**：进程刚被 `fork()` 出来，PCB 已分配但还没准备好运行
- **就绪**：一切就绪，只等 CPU 分配时间片
- **运行**：正在占用 CPU，时间片到了就被抢回就绪态
- **等待**：在等某个事件（用户输入、磁盘读完、网络响应），事件完成前不需要 CPU
- **终止**：进程结束，释放所有资源

**进程间通信（IPC）：**

进程之间是隔离的，不能直接互相读内存。IPC 机制让它们可以通信：

| 方式 | 特点 | 典型场景 |
|------|------|----------|
| 管道（pipe） | 单向，父子进程常用 | `ls | grep txt` |
| 消息队列 | 进程把消息发到一个队列，另一个读取 | 生产者-消费者模型 |
| 共享内存 | 最快！多进程映射同一块物理内存 | 高性能数据传输 |
| 信号（signal） | 发一个通知 | `kill -9` 发送 SIGKILL |
| 套接字（socket） | 支持跨网络 | 网络通信 |

</span>

## 八、CPU 调度

<span style="color: red;">

CPU 一次只能跑一个进程（单核情况），调度器决定跑谁、跑多久。

**不同系统的调度目标：**

| 系统类型 | 目标 | 例子 |
|----------|------|------|
| 批处理系统 | 吞吐量最大 | 科学计算、大数据处理 |
| 交互系统 | 响应要快，不能卡顿 | 桌面、手机 |
| 实时系统 | 必须在截止时间前完成 | 汽车刹车控制、火箭导航 |

**五种经典调度算法：**

| 算法 | 做法 | 优缺点 |
|------|------|--------|
| **先来先服务（FCFS）** | 谁先到达谁先跑 | 简单但"护航效应"严重：前面的长任务堵死后面的短任务 |
| **短作业优先（SJF）** | 选预计运行时间最短的 | 理论最优但需要预知运行时间，实际做不到 |
| **优先级调度** | 给进程优先级，挑最高的跑 | 低优先级的可能永远得不到 CPU（**饥饿**），用**老化**解决 |
| **轮转调度（RR）** | 每个进程一个时间片（10-100ms），轮流跑 | 最常用。时间片太大退化成 FCFS，太小则上下文切换开销过大 |
| **多级反馈队列** | 多个优先级队列，高优先级时间片短 | **Linux 和 Windows 实际使用的方案**，兼顾交互和批处理 |

**多级反馈队列是实际工业界的选择**——它把交互型进程（需要快速响应，比如你敲键盘）放高优先级短时间片队列，把计算密集型进程（比如视频编码）逐渐降到低优先级长时间片队列。用不完时间片的降级，等得久的升级。

</span>

## 九、进程同步与死锁

<span style="color: red;">

**竞争条件（Race Condition）：**

多个进程/线程同时访问共享数据，结果取决于执行顺序——这是多线程编程中最危险的 bug。

经典例子：两个线程各执行 `count++`（底层是三条指令：读-加-写）：

```
线程 A：读 count=5 → 加 1 → 写 count=6
线程 B：读 count=6 → 加 1 → 写 count=7  ✓ 正确

线程 A：读 count=5     →     加 1 → 写 count=6
线程 B：        读 count=5 → 加 1 → 写 count=6  ✗ 错了！结果应该是 7
```

**临界区问题的三个条件：**

1. **互斥**：同一时刻最多一个进程能在临界区里
2. **前进**：临界区空着时，想进去的进程不能无限排队
3. **有限等待**：一个进程请求进入后，等待时间有上限

**同步工具——从底层到高层：**

| 工具 | 层次 | 特点 |
|------|------|------|
| **自旋锁**（硬件原子指令实现） | 最底层 | 简单但浪费 CPU（一直在循环检查），适合临界区极短的情况 |
| **信号量**（Semaphore） | 中层 | 计数器 + P/V 原子操作，可做互斥锁也可做计数控制 |
| **管程**（Monitor） | 高层 | 封装共享数据+操作，提供条件变量。Java `synchronized` 就是管程实现 |

**经典同步问题：**

- **生产者-消费者**：缓冲区满→生产者等，缓冲区空→消费者等。用互斥信号量 + 两个计数信号量解决
- **读者-写者**：多读者可同时读，写者需独占。读者优先可能让写者饿死
- **哲学家进餐**：五人围桌，五根筷子，每人需要两根才能吃饭。方案不当会死锁

**死锁四个必要条件（缺一不可）：**

1. **互斥**：资源只能独享
2. **持有并等待**：占着资源还在等别的
3. **不可抢占**：不能强行抢进程手里的资源
4. **循环等待**：A 等 B，B 等 C，C 等 A

**死锁处理策略：**

| 策略 | 做法 | 实际应用 |
|------|------|----------|
| 预防 | 破坏四个条件之一 | 资源利用率低 |
| 避免 | 银行家算法，分配前判断安全性 | 需要预知最大需求，实际很少用 |
| 检测+恢复 | 允许死锁，定期检测，发生就杀进程或回滚 | **大多数实际 OS 的做法** |

</span>

## 十、内存管理

<span style="color: red;">

**核心问题**：多个进程怎么共享同一块物理内存？怎么保护 A 进程不偷看 B 进程的内存？

**逻辑地址 vs 物理地址：**

- **逻辑地址（虚拟地址）**：CPU 产生的地址，每个进程从 0 开始有自己独立的地址空间
- **物理地址**：内存硬件上的实际地址
- **MMU（内存管理单元）**：硬件模块，运行时把逻辑地址翻译成物理地址

**分页（Paging）——现代 OS 的内存管理核心方案：**

把物理内存切成固定大小的**页框（frame）**，把进程的逻辑地址空间切成同样大小的**页（page）**。页和页框不需要连续，可以任意映射。

```
逻辑地址：  [页号 p] [页内偏移 d]
物理地址 = 页表[p] × 页大小 + d
```

页大小通常是 **4KB**。每个进程有一个**页表**记录"第 i 页"对应"物理内存第 j 个页框"。

**TLB（快表）**：页表放在内存里，每次访问都要查一次页表（一次访存变两次），太慢。TLB 是 CPU 里的缓存，存最近用过的页表项。TLB 命中率高时，地址翻译几乎不花时间。

**分段与段页式**：分段按逻辑单元切（代码段、数据段、栈等），更贴近程序员的思维。现代 OS 基本都用**段页式**——逻辑上分段，底层用分页管理物理内存。

</span>

## 十一、虚拟内存

<span style="color: red;">

虚拟内存打破了"进程的全部地址空间都必须装进物理内存"的限制——进程只有**需要的部分**在物理内存里，其他部分暂存在硬盘上。

**三个关键好处：**

1. 进程可以比物理内存大（8GB 内存跑 N 个各 4GB 的进程）
2. 每个进程有自己的独立地址空间
3. 提高物理内存利用率——不用的页不进内存

**按需调页（Demand Paging）：**

进程启动时，OS 不分配物理页。CPU 访问地址 → 查页表 → 发现不在内存 → **缺页中断** → 从磁盘调入 → 更新页表 → 重试。

缺页率高 → 大量时间花在磁盘 I/O → 系统变慢（这就是电脑内存不够时卡顿的原因）。

**页面置换算法——内存满了踢谁？**

| 算法 | 做法 | 评价 |
|------|------|------|
| FIFO | 踢来得最早的 | 简单但可能踢掉热门页，有 Belady 异常 |
| LRU | 踢最近最少用的 | 接近最优但实现成本高 |
| **时钟算法** | 环形扫描，引用位=0就踢 | **实际 OS 最常用**，近似 LRU 但开销小 |

**时钟算法工作原理**：每个页有引用位（硬件自动置 1 表示被访问过），OS 指针循环扫描——引用位=1 就清零给第二次机会，引用位=0 就踢掉。最近用过的页最多活过完整一圈。

**颠簸（Thrashing）**：进程缺页率极高，大部分时间在等磁盘 I/O，CPU 利用率极低。原因是分配的页框太少装不下工作集。解决：要么加内存，要么减少并发数。

**写时复制（Copy-on-Write）**：`fork()` 后父子共享同一物理页，标记为只读。直到某一方尝试写入才真正复制。这对 Linux 极其重要——大部分 `fork()` 后紧跟 `exec()` 替换整个进程，过早复制等于白干。

</span>

## 十二、文件系统

<span style="color: red;">

文件系统负责把磁盘上的原始 0/1 组织成用户能理解的文件和目录。

**文件分配方式——文件在磁盘上怎么存？**

| 方式 | 做法 | 优缺点 |
|------|------|--------|
| 连续分配 | 文件占连续磁盘块 | 读写快但会有外部碎片 |
| 链接分配 | 每个数据块存指针指向下一块 | 无碎片但不能随机访问，丢一个指针全完 |
| **索引分配（inode）** | 每个文件一个索引块，存数据块列表 | **Unix/Linux 的实际方案** |

**inode 结构详解：**

每个文件对应一个 inode，存储文件的所有元信息（权限、大小、时间戳、数据块位置）：

```
inode
├─ 直接块 × 12   → 直接指向数据块（小文件：12×4KB = 48KB）
├─ 单间接块      → 指向装满指针的块（→ 4MB）
├─ 双间接块      → 两级索引（→ 4GB）
└─ 三间接块      → 三级索引（→ 4TB）
```

小文件走直接块（快），大文件自动多级索引（容量大）——这就是 inode 设计的精妙之处。

**目录**：目录本身也是一个文件，里面存的是一张表：`{文件名 → inode号}`。打开 `/home/user/foo.txt`，OS 从根目录 inode 开始逐层查。

**日志文件系统（Journaling）**：写数据前先把"我要做什么"记入日志区。断电后：
- 已完成的事务 → 重放
- 半截的事务 → 丢弃

常见日志文件系统：ext3/ext4 (Linux)、NTFS (Windows)、APFS (macOS)。

</span>

## 十三、I/O 系统

<span style="color: red;">

**I/O 三种方式——从低效到高效：**

| 方式 | 做法 | CPU 利用率 |
|------|------|------------|
| **轮询** | CPU 一直循环问设备"好了没" | 几乎 0%，CPU 全浪费在等 |
| **中断驱动** | 设备干完活发中断，CPU 期间干别的 | 大幅提升 |
| **DMA** | CPU 发一次命令，DMA 控制器自己搬数据，搬完发信号 | **最优**，CPU 完全解放 |

DMA（直接内存访问）对于大块数据传输极其重要——比如磁盘读写、网络传输。CPU 只发一条命令，后面的数据搬运全由 DMA 控制器完成。

**磁盘调度（针对机械硬盘）：**

| 算法 | 做法 | 评价 |
|------|------|------|
| FCFS | 按请求顺序服务 | 公平但磁头乱跳 |
| SSTF | 选离当前位置最近的 | 快但远的请求可能饿死 |
| **电梯算法（SCAN）** | 磁头沿一个方向扫到底再反向 | 常用，类似电梯运行方式 |
| **C-LOOK** | 单向服务到头直接跳回 | Linux 默认方案 |

现代 SSD 没有移动部件，不需要磁盘调度——I/O 调度转向合并相邻请求和优化优先级。

</span>

## 十四、安全与保护

<span style="color: red;">

**保护 vs 安全：**
- **保护**：防止进程访问不该访问的资源（内部机制）
- **安全**：防止外部攻击（外部威胁）

**Unix 权限模型：**

每个文件用 9 个位控制权限：`rwx rwx rwx`（属主-同组-其他人）

```bash
-rw-r--r--  1 root root 4096 May 1 12:00 file.txt
# owner: rw-    (可读可写)
# group: r--    (只读)
# others: r--   (只读)
```

`rwx` = read(4) + write(2) + execute(1)，`chmod 755` = rwxr-xr-x。

**常见安全威胁：**
- **缓冲区溢出**：输入超过数组大小，覆盖返回地址，跳转到恶意代码
- **SQL注入**：拼接用户输入到 SQL 查询，绕过验证

**防御：**
- DEP/NX 位禁止数据页执行代码
- ASLR 把栈和堆地址随机化
- Canary 值：返回地址前放随机值，被改则崩溃

</span>

## 十五、WSL2 实操命令速查

<span style="color: red;">

以下是在 WSL2/Linux 环境中练习操作系统概念的常用命令：

```bash
# 进程
ps aux                    # 查看所有进程
top                       # 实时监控（按 q 退出）
kill -9 PID               # 强杀进程
nice -n 10 cmd            # 以低优先级运行

# 内存
free -h                   # 内存使用概览
cat /proc/meminfo         # 详细内存信息
cat /proc/PID/maps        # 查看某个进程的内存映射

# 系统调用
strace ls                 # 看 ls 用了哪些系统调用
strace -c cmd             # 统计所有系统调用的次数和耗时

# 文件
lsof                      # 列出所有打开的文件
ls -li                    # 查看 inode 号
stat file.txt             # 查看文件详细信息（inode、块数等）

# 磁盘
df -h                     # 磁盘使用情况
iostat -x 1               # 磁盘 I/O 实时统计

# 其他
uname -a                  # 内核版本
dmesg | tail -20          # 最近的内核消息
```
